# 지원자 스펙 Gap 보고서 Agent와 근거 수집 파이프라인

이번 변경은 결정론적 자격·능력 Gap 계산 위에 Bedrock Knowledge Base와 Q-Net 근거를 수집하고, Strands Agent 또는 결정론적 fallback으로 JSON/Markdown 보고서를 만드는 기능을 추가한다. 자격그룹의 `하나 이상` 판정과 능력명/NCS 코드 매칭은 규칙 엔진에서 처리하고, Agent 결과는 validator가 핵심 판정 필드와 대조한다. RAG 변환, 선택적 S3 저장, Q-Net 캐시, SAM 리소스까지 한 번에 추가되며 기존 CrewMate 응답 스키마와 프런트 lockfile도 함께 바뀐다. 치명적 문제는 확인되지 않았지만, 배포 전 수정해야 할 높은 우선순위 문제가 남아 있다.

**Watch for:** **confirmed** `persistReport=false`에서도 Q-Net 캐시 DDB 쓰기가 발생할 수 있고 Agent의 Q-Net 도구도 실제로는 쓰기 동작을 한다; **confirmed** 캐시 장애가 partial fallback을 우회해 전체 요청을 실패시킨다; **confirmed** 자유 입력 일부가 Bedrock Agent 프롬프트로 전달되어 PII 차단이 완전하지 않다; **confirmed** authoritative 검증이 원래의 검토·한계 경고와 재시도 후 근거 완전성을 보호하지 않는다; **confirmed** 기본 KB 필터와 생성 데이터 상태가 달라 기본 배포에서 근거가 검색되지 않는다.

## High-level view

결정론적 계산은 자격그룹 단위로 `하나 이상`을 평가하므로 한 자격으로 그룹이 충족되면 나머지 자격을 부족 항목으로 만들지 않는다. 능력은 정규화한 능력명 또는 NCS 코드의 정확 매칭으로 판정하며, 이 경로는 테스트로 고정되어 있다.

Agent가 변경하지 못하도록 보호하는 핵심 판정 필드는 비교되지만, 구조화 분석에 포함된 `limitations`와 `humanReviewItems`는 비교 대상이 아니다. 첫 Agent 결과에는 근거 누락 검사를 적용하면서 결정론적 근거를 제공한 두 번째 결과에는 같은 검사를 생략해, 근거 실패나 검토 필요 상태가 최종 보고서에서 사라질 수 있다.

등록된 Agent 도구는 KB 조회와 Q-Net 조회 두 개뿐이다. 다만 Q-Net 조회가 내부적으로 DynamoDB 캐시를 갱신하므로 읽기 전용 보장은 성립하지 않으며, 이 쓰기는 `persistReport`와 무관하게 실행된다.

KB와 Q-Net 어댑터는 원격 실패를 상태 객체로 바꾸지만 Q-Net DynamoDB 캐시의 읽기·쓰기는 예외를 정규화하지 않는다. 캐시 권한, throttling, 일시 장애가 발생하면 fallback 보고서 대신 API 500으로 끝난다.

입력 모델의 자유 문자열인 `targetSpecialty`와 미매칭 자격 입력은 구조화 결과에 남아 Bedrock 모델 프롬프트로 전달된다. KB query에만 적용한 정규식 PII 검사는 이 모델 호출 경로를 보호하지 않는다.

CSV와 `.metadata.json`은 Bedrock record-based CSV 구조에 맞는다. 그러나 기본 `KB_REVIEW_STATUS=검토완료`와 현재 생성 자산의 상태가 일치하지 않아 기본 구성에서는 해당 문서가 모두 필터링된다.

SAM은 Cognito 기본 authorizer, 비공개 S3, KMS 출력 암호화, prefix 단위 S3 권한을 적용한다. 반면 두 DDB 테이블에 전체 CRUD를 부여하고 KB IAM ARN의 리전은 실제 조회 리전이 아닌 스택 리전을 사용해 최소권한과 교차 리전 구성에 틈이 있다.

기존 CrewMate의 worker self 응답에서 `dispatched_count`를 숨기는 변경은 기존 노출 정책 및 통합 테스트와 일치한다. 백엔드 전체 테스트와 SAM lint는 통과했지만, 프런트 빌드는 Tailwind/Vite native 단계의 Windows 예외로 종료되어 lockfile 변경의 회귀 여부는 검증하지 못했다.

## 구조화 판정은 고정되지만 경고와 근거 완전성은 고정되지 않음

**confirmed** `backend/spec_report/validator.py:15-28`의 authoritative 필드 목록에서 `StructuredGapAnalysis.human_review_items`와 `.limitations`가 빠져 있다. Agent가 세부 작업 미지정 경고나 구조화 규칙 버전 한계를 삭제해도 검증이 성공하므로, 두 필드를 원본의 순서 보존 부분집합으로 강제하거나 Agent 보고서에 결정론적 원본 경고를 병합해야 한다.

**confirmed** `backend/spec_report/orchestrator.py:147-150`은 첫 Agent 응답에 `missing_evidence_items()`를 적용하지만, 결정론적 근거를 넣어 재호출한 응답은 `:160-165`에서 구조 검증만 한 뒤 채택한다. 두 번째 응답이 KB/Q-Net 결과와 실패 limitation을 생략해도 fallback이 실행되지 않으므로 두 pass 모두 같은 evidence-plan coverage 검증을 적용하고, 실패 상태도 해당 item의 limitation 또는 human-review 항목으로 남도록 검사해야 한다.

## Q-Net 조회의 숨은 쓰기와 `persistReport=false` 계약 위반

**confirmed** `backend/spec_report/orchestrator.py:176`의 저장 gate는 보고서 S3와 jobs DDB 저장만 막고, 그 전에 실행되는 `backend/spec_report/qnet.py:214-216`은 SUCCESS, NAME_MISMATCH, UNAVAILABLE 결과를 모두 `self.cache.put()`으로 기록한다. 따라서 `persistReport=false`의 S3/jobs 미쓰기는 성립하지만 DDB 미쓰기 계약은 깨지며, Agent에 등록된 도구가 KB/Q-Net 두 개뿐이어도 `fetch_qnet_qualification`은 동작 수준에서 쓰기 도구다.

요청 단위 read-only cache 모드를 추가해 `persistReport=false`에서는 `put`을 생략하거나, cache 갱신을 Agent 도구 밖의 명시적 persistence 단계로 분리해야 한다. 테스트도 online Q-Net 성공/실패와 fake DDB table을 사용해 false일 때 `put_item`이 0회인지 검증해야 한다; 현재 테스트는 offline `FakeStorage`만 확인하고 Agent registry 테스트도 상수 tuple만 비교한다.

## 캐시 장애가 partial fallback 대신 전체 실패를 만듦

**confirmed** Bedrock KB와 Q-Net HTTP 예외는 `UNAVAILABLE`로 변환되지만 `DynamoQualificationCache.get()`과 `.put()`의 DDB 호출은 예외를 잡지 않고, `SpecReportService._collect()`도 item별 예외 경계를 두지 않는다. 캐시 throttling, IAM 오류, table 장애가 `generate()` 밖으로 전파되어 Lambda 500으로 끝나므로, cache read 실패는 miss로, write 실패는 Q-Net evidence 유지와 limitation 추가로 downgrade하고 한 source 실패가 다른 evidence 수집을 중단하지 않게 해야 한다.

## Bedrock Agent 프롬프트의 PII 우회 경로

**confirmed** `backend/spec_report/report_agent.py:103-108`은 `structured.model_dump()` 전체를 Bedrock 모델에 전달한다. 길이만 제한되는 `targetSpecialty`와 미정규화 자격 원문 `normalizedCertifications[].inputName`에 전화번호나 이메일이 섞이면 KB-query PII 검사와 무관하게 모델 호출로 전송된다. Agent 전용 최소 DTO에서 raw `inputName`과 불필요한 자유 입력을 제거하고, 남는 자유 문자열에는 동일한 PII detector를 fail-closed로 적용해야 한다.

## RAG CSV는 유효하지만 기본 review gate가 모든 근거를 제외함

Bedrock record-based CSV sidecar는 `search_text`를 content field로, trade·document type·NCS code·review status 등을 metadata field로 선언해 요청한 CSV/metadata 형식 문제는 확인되지 않았다.

**confirmed** 생성된 자격·능력 규칙은 `원본 상태 미표기`, RAG 검색 문서는 `사람 검토 전 자동 추천`인데 `template.yaml`과 retriever 기본값은 `검토완료`다. 현재 자산을 그대로 업로드하면 default filter가 빈 결과를 만들므로 배포 전에 검토 완료 레코드를 생성하고 manifest에 승인 상태를 기록하거나, 의도한 fail-closed 상태라면 배포 gate와 운영 문서에서 KB가 partial 상태임을 명시해야 한다.

## SAM의 저장소 경계와 리전 비대칭

신규 API는 `CrewMateApi`의 Cognito default authorizer를 상속하고, S3 public access block·output KMS 암호화·`rules/*` 읽기와 `reports/*` 쓰기 분리가 적용된다.

**confirmed** `template.yaml:693-694`는 qualification cache와 jobs table 모두에 `DynamoDBCrudPolicy`를 부여해 실제 필요한 cache GetItem/PutItem과 jobs PutItem보다 넓다. DeleteItem, UpdateItem, Scan 등을 제거한 명시적 statement로 축소하고, DDB 무쓰기 요구를 정책으로 강제하려면 jobs 저장을 별도 함수로 분리해야 한다.

**confirmed** retriever는 `REPORT_MODEL_REGION`으로 KB client를 만들지만 IAM resource는 `arn:aws:bedrock:${AWS::Region}:...`을 사용하므로 두 값이 다르면 권한 거부된다. ARN에 `${ReportModelRegion}`을 쓰거나 KB 리전을 별도 파라미터로 분리하고, 모든 리전의 foundation model/inference profile에 열린 invoke 권한도 선택한 모델 ARN으로 좁혀야 한다.

## 기존 CrewMate 호환성과 검증 범위

worker self 응답의 `dispatched_count` 비노출은 파일 상단 정책과 `tests/test_v2_flow.py`의 기존 assertion에 맞아 확인된 CrewMate API 회귀는 없다. `python -m pytest -q` 43개, Python compileall, `sam validate --lint`, `git diff --check`는 성공했다.

**confirmed** 보고서 기능과 무관한 `frontend/package-lock.json` 변경이 같은 diff에 포함되어 있고, `npm run build`는 Vite/Tailwind 변환 중 Windows native 예외 `-1073740791`로 종료됐다. base lockfile에도 같은 optional native 패키지가 있어 원인을 이번 변경으로 단정할 수는 없지만, lockfile churn은 분리하거나 깨끗한 설치 CI의 빌드 성공 후 포함해야 한다.
